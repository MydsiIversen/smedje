# Smedje — Claude Code project context

## What we're building

Smedje (github.com/smedje/smedje, smedje.net) is a Go open-source toolkit
for forging every key, ID, cert, and config a developer or network engineer
generates from scratch. It ships as:

- A single static binary CLI: `smedje`
- An importable Go library: `pkg/forge/...`
- (Eventually) an embedded web frontend with a wizard for designing
  multi-site networks and outputting working VyOS / RouterOS / OPNsense
  / Linux configs

The name is Danish for "smithy." Smedje exists to replace the dozen
single-purpose binaries you juggle whenever you need to generate
artifacts from scratch — UUIDs, Snowflake IDs, SSH keys, TLS certs,
WireGuard configs, IPsec PSKs, TOTP secrets, MAC addresses, passwords,
and so on. One binary, one mental model, consistent output.

Smedje is a personal hobby project. Don't reference any employer or
client in code, comments, commits, docs, or marketing copy.

## What Smedje is *not*

- Not a key management system (no key storage, no rotation)
- Not a PKI service (we generate certs, we don't run a CA)
- Not a secrets manager (no encryption-at-rest, no vault)
- Not an AI tool (zero LLM dependencies, ever)
- Not a teaching tool (assumes you know what a TLS cert is)

If a feature request blurs these lines, push back.

## Stack

- Go 1.23+
- Cobra for CLI structure
- crypto/* from stdlib first; golang.org/x/crypto only when stdlib doesn't
  cover the primitive
- wgctrl-go for WireGuard
- Standard testing package; no testify or other test frameworks
- For Phase 2+: Next.js 15 + Mantine for the web frontend, embedded via
  go:embed; tRPC over a thin Go HTTP server

No surprise dependencies. Every new import in go.mod needs a one-line
justification in the PR.

## Architecture rules

1. **Plugin pattern.** Every generator implements a Generator interface
   in pkg/forge/generator.go and self-registers in init(). Adding a new
   generator means adding ONE file under pkg/forge/<category>/<name>.go
   and it shows up in the CLI, library, benchmark suite, and (eventually)
   web UI automatically.

2. **CLI is a thin wrapper.** All real logic lives in pkg/forge/. The
   cmd/smedje/ package translates Cobra args into Generator.Options and
   renders the result. People should be able to import a single generator
   without pulling in Cobra.

3. **Output is consistent.** Default output is human-readable. --json
   switches to structured. --quiet strips everything but the value.
   Never mix modes.

4. **No global state.** Pass context.Context everywhere. Pass options
   structs explicitly. The only exception is the generator registry,
   which is necessarily global and is the one place I want it.

5. **Crypto rules.** Smedje uses well-vetted primitives only. Never invent
   crypto. Never roll our own random. Use crypto/rand for entropy. When in
   doubt, follow what age, step-cli, or wgctrl-go do.

6. **Performance matters but readability wins ties.** ID generators should
   benchmark well (target: 10M ops/sec/core for UUIDv7) but if a 5%
   performance gain costs readability, skip it.

## Naming conventions

- Generator names match their CLI command: `uuid v7`, `snowflake`, `ssh
  ed25519`, `tls self-signed`, `wireguard keypair`, `totp`, `password`,
  `mac`. Hyphens, not underscores. Kebab-case for multi-word.
- Go package names follow stdlib style: short, lowercase, no underscores.
- Errors: `ErrXxx` for sentinel errors, wrap everything else with %w.
- Test files: standard Go _test.go.

## Voice for code comments and docs

Plain English. Concrete. No marketing words. Comments explain *why* not
*what*. Doc comments on every public symbol, full sentences ending in
periods.

Bad:  // Generate creates a powerful new UUID
Good: // Generate returns a UUIDv7 with the timestamp set to time.Now().
      // The random component uses crypto/rand and is 74 bits per RFC 9562.

## Working style for this project

- Show progress incrementally. Build → test → review → continue.
- After every meaningful step, summarize what changed and what's next in
  3-5 bullets.
- Flag every default you pick where someone might reasonably disagree
  (key sizes, password length defaults, RNG choices). Don't ask permission
  for obvious defaults; do call them out.
- Prefer one well-documented file over three clever ones.
- When you hit something I should know about (a stdlib quirk, a security
  consideration, a deferred decision), surface it explicitly. Don't bury
  it.
- Run go test, go vet, and gofmt before claiming a step is done.

## What we're NOT doing yet (deferred to later phases)

- Web frontend / wizard
- Network design (sites, VPNs, topology)
- Multi-vendor config rendering (VyOS, RouterOS, OPNsense, Linux)
- explain mode (paste anything, get a breakdown)
- Profile system (YAML-defined sequences)
- TUI mode (Bubble Tea)
- --paranoid mode (real implementation; entropy package has stubs)
- Cloud-init, k8s, Ansible/Terraform output

If you find yourself wanting to add any of these, stop and remind me.
The point of v0.1 is a tight, ship-able CLI library foundation.
